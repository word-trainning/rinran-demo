# 技術要件 — 出前中華りんらん リニューアル + 顧客管理システム

最終更新: 2026-05-02
対象: ホームページリニューアル (`rinran-demo`) + 顧客管理システム (`rinran-crm`)

---

## 1. 全体構成

本案件は **2つのシステム** で構成されます。

| システム | 役割 | 公開範囲 | リポジトリ |
|---|---|---|---|
| **ホームページ** | 集客・問い合わせ獲得 (法人仕出し主役) | 一般公開 | `word-trainning/rinran-demo` |
| **購買傾向分析システム (CRM)** | 過去注文の **業種・用途・金額の集約と傾向分析** (※個人情報は保持しない) | 店舗内クローズド | `word-trainning/rinran-crm` |

> **重要な前提**: 仕出し弁当系ポータルサービス (yuizen / 出前館 / プレミアムスタイル等) の利用規約では
> ポータル経由で得た顧客情報を用いた直接営業 (DM・電話・メール勧誘) が禁止されている。
> 本 CRM はこのリスクを根本から排除するため、**個人名・電話番号・メールアドレス・住所・担当者名は一切登録せず**、
> **各業界の購買傾向把握 (集計・分析) 専用** として運用する。

両者は **運用上は完全分離**。CRMは店舗外からアクセス不可。提案資料上のみ、ホームページから1クリックで遷移できる導線を設置。

### 2.3 機能要件

- **問い合わせ導線**: 電話 / LINE / 見積もりフォームの3チャネル
- **見積もりフォーム**: 法人情報・配達条件・希望日時の入力 → **Google スプレッドシート自動記録 + 店舗 Gmail 通知** (Google フォーム + GAS、有料 SaaS を使わない月額ゼロ円運用。Phase 2 で個人情報を含まない業種・用途データのみ CRM へ連携)
- **メディア掲載コーナー**: TV番組5本の実績表示
- **ポータル接続表示**: 7社 (yuizen / プレミアムスタイル / 出前館 / dデリバリー / 楽天デリバリー / Uber Eats / LINEデリマ) を併設
- **モバイル最適化**: スマホ縦持ちで全コンテンツ閲覧可

### 2.4 セキュリティ

- HTTPS必須 (デモ段階: GitHub Pages 自動対応 ／ 本番: 既存サーバの SSL 設定を継続、必要に応じて Let's Encrypt で補強)
- 関係者向けデモはパスワードゲート (`gate.html` + `gate.js`、SHA-256ハッシュ照合)
- noindex/nofollow メタタグでデモ段階の検索エンジン露出を抑止
- お問い合わせ送信時の reCAPTCHA は本実装時に検討

## 3. 購買傾向分析システム (CRM)

### 3.1 機能要件

| 機能カテゴリ | 内容 |
|---|---|
| **取込パイプライン** | 注文書を撮影/PDF添付 → 自動文字読取 → **個人情報欄を自動マスク** → 業種・用途・金額・要望タイプのみ抽出 → 承認 のフロー。原本画像は抽出完了後に即時破棄 |
| **直接入力** | 電話注文・OCR失敗時の手入力フォーム (個人情報フィールドは入力欄自体を表示しない) |
| **業種マスタ** | 業種カテゴリ × 会社規模カテゴリ × 用途 単位での匿名集約。個人・会社名そのものは保持せず、業種コードに変換して破棄 |
| **注文履歴 (匿名)** | 業種・用途・金額帯・月日・要望タイプでの絞り込み (個人特定は不可能な設計) |
| **要望管理** | 8タイプ (アレルギー/食事制限/時間指定/配達方法/用途/盛付/支払特記/駐車) を構造化保存。要望文の自由記述は保存せず、タイプ分類のみ |
| **分析ダッシュボード** | 7軸 (月次/支払/業種ランク/獲得時期/売上の流れ/24区マップ/要望) — すべて業種×期間の集計値 |
| **状態管理** | 下書き (draft) / 承認済み (approved) / 取り消し (rejected) の3状態で誤集計防止 |

> **明示的に保持しない情報**: 氏名、担当者名、電話番号、メールアドレス、配達先住所 (区コードまで丸める)、会社名、注文書原本画像。
> これらは取込フローで自動破棄し、データベースに到達させない。

### 3.2 技術構成

| Webフレームワーク | FastAPI | 0.115 |
| ASGIサーバ | Uvicorn | 0.32 |

| セッション署名 | itsdangerous | 2.2 |
| マルチパート | python-multipart | 0.0.12 |
| データベース | SQLite (Phase 1) → PostgreSQL (Phase 2) | 17 (本番昇格時) |

#### フロントエンド
| 区分 | 採用技術 |
|---|---|
| マークアップ | HTML5 |
| スタイリング | Tailwind CSS (CDN) |
| インタラクション | Vanilla JavaScript |
| グラフ・地図 | Apache ECharts 5.5 (CDN) |
| 地理データ | 大阪市24区 GeoJSON (オフラインバンドル、約400KB) |
| 画像処理 | ブラウザ標準 (camera capture / multiple file upload) |

#### OCR (文字読み取り)
| 区分 | 内容 |
|---|---|
| アーキテクチャ | プロバイダ抽象層で差し替え可能 |
| Phase 1 (現在) | Stub (擬似データ生成、UI/フロー検証用) |

| Phase 2以降 | (任意) クラウドOCR or ローカルLLM補完 |
| 要望抽出 | ルールベースキーワードマッチ (8タイプ × 約60キーワード) |

### 3.3 データ構造

| テーブル | 主用途 | 行数規模 (本番想定) | 保持しないカラム |
|---|---|---|---|
| `industry_segments` | 業種×規模カテゴリのマスタ (匿名) | 50〜200カテゴリ | 会社名・担当者名・連絡先 |
| `orders` | 注文ヘッダ (業種コード参照のみ) | 1,000〜10,000件 (年間) | 顧客ID・氏名・住所 (区コードのみ保持) |
| `order_items` | 注文明細 | 5,000〜50,000行 | — |
| `order_requirements` | 要望タイプ集計 (自由記述は保存しない) | 1,500〜15,000行 | 自由記述本文 |
| `import_batches` | 取込ロット履歴 | 数百〜数千 | 原本画像 |
| `audit_log` | 操作監査 | 数千〜数万 | — |
| `schema_migrations` | マイグレーション履歴 | 数件 | — |

マイグレーションは `db/migrations/_manual/00NN_*.sql` に手書きSQLとして配置、起動時に未適用分を自動実行。

#### CRM のサーバ要件 (既存サーバ同居 or 追加 VPS どちらでも適用)

## 5. セキュリティ・コンプライアンス

| 項目 | 対応 |
|---|---|
| **個人情報保護 (根本対策)** | **個人情報を保持しない設計**。氏名・電話・メール・住所・担当者名・会社名はそもそもDBに格納しない (取込時に自動破棄)。漏洩リスク自体を発生させない |
| **仕出し弁当系ポータル規約遵守** | ポータル経由顧客への直接営業を構造的に不可能にする。連絡先が DB に存在しないため、DM・架電リスト作成が物理的にできない |
| **用途限定** | 取得データは「各業界の購買傾向把握 (集計・分析)」専用。特定顧客への営業活動には一切使用しない (運用規約に明文化) |
| 認証 | 単一管理者パスワード (Phase 1) → 多ロール展開 (Phase 2) |
| セッション | HMAC署名Cookie、HttpOnly/SameSite |
| クローズド運用 | 外部メール送信機能を実装しない (誤送信防止) |
| データ外部送信 | 一切なし (Phase 2 OCRもローカル実行) |
| 監査ログ | view/edit/export/login をDB保存 |
| アクセス制御 | Phase 2 はIP制限+VPN推奨 |
| 定期バックアップ | 日次 + 月次フルバックアップ別保管 |
| 災害復旧 | スナップショット2世代以上保持、復旧手順書整備 |
| 注文書原本 | OCR抽出完了後に即時破棄。バックアップにも残さない |

### Phase 1 (完了済み)

- 架空データでの動作検証

### Phase 2 (本番昇格)
| ステップ | 内容 | 想定期間 |
|---|---|---|
| 2-A | リニューアルHP本実装 (既存写真の活用・最適化、新規撮影なし) | 2〜3週間 |
| 2-B | CRM デプロイ・HTTPS化 (既存サーバ同居 or 追加VPS) | 1週間 |

| 2-E | 店主オンボーディング・操作研修 | 2〜3日 |
| 2-F | 本番運用開始 | — |

### Phase 3 (連携拡張・低コスト構成)
- **ホームページ見積もりフォーム連携の基本方針**: 有料 SaaS (kintone 等) は採用せず、**Google フォーム + Google スプレッドシート + Google Apps Script (GAS)** の無料枠で月額ゼロ円運用
  - 見積もりフォーム送信 → スプレッドシート自動行追加 → GAS から店舗 Gmail へ通知メール
  - 店主は普段使いの Gmail / スマホでスプレッドシートを開くだけで確認可能
  - 個人情報 (氏名・連絡先・住所) はスプレッドシート側に留め、CRM へは **業種・用途・金額帯のみ** をエクスポート
- LINE公式アカウント連携 (問い合わせ集約) — 同じく無料枠 (LINE Messaging API 月1,000通まで無料) で構築
- 直接決済導入 (Square / AirPay 等の検討) — いずれも月額0円のプランから開始

### Phase 4 以降 (任意)
- 法人会員制度・定例配達契約管理
- 撮影現場向け定額プラン管理
- 月次経営レポート自動生成 (PDF/メール送信)

## 8. 運用体制

### 店主 (主担当)
- 注文書の取込 (毎日)
- 確認待ちキューの承認 (毎日)
- 週次の分析画面チェック
- バックアップ確認 (月次)

### 担当者 (技術担当)
- システム稼働監視
- マイグレーション・機能追加
- 障害対応・パスワードリセット
- OCR精度の継続改善

### 引き継ぎ可能性
本システムは**店主単独運用**を最優先に設計。担当者が不在になった場合も、操作マニュアル (`docs/OPERATIONS.md`) と起動手順のみで継続運用可能。

## 9. リソース・依存

### コードベース規模 (Phase 1完了時点)
- ホームページ: HTMLページ11枚 + CSS約2,500行 + JS小規模
- CRM: Pythonコード約2,500行 (ルート4ファイル + OCR層 + DB層)
- 静的アセット: 画像合計約4MB / GeoJSON 400KB
- マイグレーションSQL: 3ファイル (initial / import_pipeline / requirements)

### 外部サービス依存 (有料 SaaS は採用せず、無料枠で運用)
| サービス | 用途 | 月額コスト | 必須度 |
|---|---|---|---|
| Google Fonts | Webフォント配信 | 無料 | 高 (フォールバック対応済) |
| jsDelivr CDN | Tailwind CSS / ECharts 配信 | 無料 | 中 (オフライン化可能) |
| GitHub Pages | デモホスティング (Phase 1のみ。本番は既存サーバへ移設) | 無料 | 低 |
| 既存サーバ (店舗契約) | **本番ホスティング (HP本体)。新規契約・サーバ移行は行わない** | 店舗の現契約料金 (新規発生なし) | 高 |
| Let's Encrypt | SSL証明書 (Phase 2) | 無料 | 高 |
| **Google フォーム + スプレッドシート + GAS** | **見積もりフォーム受付・店舗通知 (Phase 2-)** | **無料 (Google 個人アカウント枠で完結)** | 高 |
| Gmail | 店舗側通知の受信 | 無料 | 高 |
| (任意) UptimeRobot | 死活監視 | 無料プランあり | 低 |

**有料 SaaS を採用しない理由**: 店主おひとりの運用継続性を最優先とし、月額固定費を発生させない。
将来、件数が増えて Google 無料枠を超える段階で初めて有料化を検討する (現状の問い合わせ件数では無料枠で十分余裕)。

**ベンダーロックインなし** — 既存サーバ・既存ドメインを継続利用するため切替コスト発生なし。データはすべてSQLite/PostgreSQLの標準形式で保管、将来の移行も自由。
スプレッドシートも CSV エクスポートで他システムに移行可。

## 10. 用語解説 (店主向け)

| 用語 | 平易な意味 |
|---|---|
| OCR / 自動読取 | 画像から文字を読み取る機能 |

| マイグレーション | データベース構造を新しくする作業 |
| API | 画面とデータベースをつなぐ部品 (店主が直接触ることはない) |
| GeoJSON | 地図の境界線データのファイル形式 |
| systemd | Linuxサーバで自動起動を管理する仕組み |
| HTTPS | 通信を暗号化して安全にする方式 (鍵マークが付く) |
| VPS | 借りる仮想専用サーバ |
